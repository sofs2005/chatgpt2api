# 用户计费与额度分配需求

## 1. 背景

当前系统已经具备三类和“额度”相关的能力：

- 账号池额度：ChatGPT 账号导入后会刷新账号类型、状态、图片剩余额度和恢复时间，用于选择可用上游账号。
- 用户限流：普通用户有全局默认创作并发额度和默认 RPM 限制，管理员不受影响。
- 用户统计：用户管理页已经展示调用次数、成功/失败次数、额度消耗和近 14 天调用曲线。

但系统尚未支持管理员给不同用户分配本地可消费额度。现有“账号池额度”只能代表上游账号可用量，不能约束某个本地用户或 Linuxdo 用户可使用多少本地额度；现有“并发/RPM”只限制速度，不限制总量。

本需求目标是在现有用户管理、创作任务、OpenAI 兼容接口和日志统计之上，增加一套本地用户计费与扣减机制。该机制需要明确支持两种计费类型：

- 标准（余额）制度：用户拥有可消费余额，成功产出后按量扣减余额，不按周期自动恢复。
- 订阅（配额）制度：用户在当前订阅周期内拥有固定配额，成功产出后扣减当期配额，周期到期后按规则重置。

本文中的“计费”是本地额度核算概念，不代表支付、订单、充值渠道或财务结算能力。

## 2. 目标

- 管理员可以为每个普通用户选择计费类型：标准余额制或订阅配额制。
- 标准余额制用户提交创作请求时，系统按本地余额判断是否允许继续。
- 订阅配额制用户提交创作请求时，系统按当前周期剩余配额判断是否允许继续。
- 创作成功后按实际产出扣减用户余额或当期配额。
- 用户可以在个人资料或创作页看到自己的计费类型、可用余额/剩余配额和当前限制状态。
- 管理员可以在用户管理页查看、调整、重置用户计费状态，并看到使用情况。
- 本地用户计费状态与账号池上游额度保持分离：用户计费管“谁能用多少”，账号池额度管“当前有哪些上游账号可执行”。

## 3. 非目标

- 不改变上游 ChatGPT 账号额度刷新逻辑。
- 不把用户余额/配额和账号池额度合并成同一个池。
- 不引入支付、订单、在线充值、退款、发票、账单或财务结算能力。
- 不实现多租户组织额度、部门额度或代理商分销。
- 不要求对历史日志重新扣减余额或配额；历史日志仅用于展示统计。
- 不改变 RBAC 权限模型，只新增必要的计费管理权限项。
- 不实现纯文本 token 级计费；纯文本聊天不扣减图片计费单位。

## 4. 术语

- 本地计费状态：系统为每个用户维护的本地余额或订阅配额状态，不等同于上游账号池额度。
- 计费类型：用户当前使用的扣减制度，取值为 `standard` 或 `subscription`。
- 标准（余额）制度：用户拥有一个可消费余额池；成功任务直接扣减余额；余额不会因为自然周期自动恢复。
- 订阅（配额）制度：用户拥有一个当前周期配额；成功任务扣减当期已用配额；周期到期后按配置重置。
- 计费单位：一次成功产出的图片按图片张数计费；图片编辑同样按成功产出张数计费；`/v1/chat/completions` 或异步对话任务只要进入图片生成模式并成功产出图片，也按成功产出张数计费；纯文本聊天不消耗图片计费单位。
- 余额：标准余额制下用户当前剩余的可消费点数。
- 可用余额：标准余额制下用户当前剩余的可消费点数，下限为 0。
- 配额上限：订阅配额制下当前周期可消费的基础点数。
- 已用配额：订阅配额制下当前周期内已成功消费的点数。
- 手动配额调整：订阅配额制下管理员在当前周期临时增加或扣减的点数。
- 剩余配额：`配额上限 + 手动配额调整 - 已用配额`，下限为 0。
- 无限额度：不受本地余额或订阅配额约束，但仍受 RBAC、账号池可用性、并发和 RPM 限制约束。
- 额度不足冻结：用户因余额或配额不足被拒绝继续提交消耗计费单位的任务，不等同于账号禁用。
- 订阅周期：订阅配额自动恢复或重置的时间范围，例如每日、每周、每月。

## 5. 用户故事

- 作为管理员，我希望给不同用户设置不同的计费类型，区分按量余额用户和订阅配额用户。
- 作为管理员，我希望新用户自动获得默认计费状态，减少手动配置成本。
- 作为管理员，我希望能临时增加或扣减某个用户的余额或当期配额，用于处理补偿、测试或违规使用。
- 作为管理员，我希望能查看用户余额/配额消耗和剩余情况，判断是否需要扩容账号池或调整分配策略。
- 作为普通用户，我希望在提交任务前知道自己是否还有可用余额或剩余配额，避免任务排队后才失败。
- 作为 API 调用方，我希望余额或配额不足时收到明确、稳定的错误码，便于客户端处理。

## 6. 功能需求

### 6.1 计费类型配置

系统需要支持以下配置层级：

- 全局默认计费类型：新建普通用户或首次出现的 Linuxdo 用户使用该默认计费类型。
- 全局默认标准余额：默认计费类型为标准余额制时，新用户获得的初始余额。
- 全局默认订阅配额：默认计费类型为订阅配额制时，新用户获得的周期配额。
- 单用户计费覆盖：管理员可以为某个用户设置独立计费类型和对应参数。
- 无限额度开关：管理员可以将指定用户设置为无限额度。
- 订阅周期：订阅配额制支持每日、每周、每月；标准余额制不自动周期重置。
- 周期锚点：每日按本地日期 00:00 重置；每周默认周一 00:00；每月默认每月 1 日 00:00。

默认策略：

- 管理员不受本地用户计费限制。
- 普通用户默认继承全局默认计费类型和默认数值。
- 如果未配置全局默认计费类型，默认使用标准余额制。
- 如果未配置全局默认标准余额，默认余额为 0，不表示无限额度。
- 如果未配置全局默认订阅配额，默认配额为 0，不表示无限额度。
- 用户设置为无限额度时，不参与本地余额不足或配额不足判断。
- 已禁用用户仍按现有认证逻辑拒绝，不进入计费判断。

计费类型切换要求：

- 管理员可以将用户从标准余额制切换为订阅配额制，必须设置新的配额上限、订阅周期和切换原因。
- 管理员可以将用户从订阅配额制切换为标准余额制，必须设置新的余额或保留已有标准余额，并填写切换原因。
- 切换计费类型不删除历史消费记录和调整记录。
- 非当前计费类型的状态可以保留用于审计和回切，但不得参与当前请求预检和扣减。

### 6.2 计费消费规则

可计费任务范围：

- 文生图：按成功返回的图片 URL 数量扣减。
- 图生图：按成功返回的图片 URL 数量扣减。
- OpenAI 兼容图片接口：按成功返回的图片 URL 数量扣减。
- OpenAI 兼容对话生图接口 `/v1/chat/completions`：当 `model` 为图片模型，或 `modalities` 包含 `image`，或响应中实际包含图片产出时，按成功产出的图片数量扣减。
- 异步图片任务：按任务最终成功产出的图片数量扣减。
- 异步对话任务 `/api/creation-tasks/chat-completions`：按任务最终成功产出的图片数量扣减。
- 失败任务、取消任务、超时任务不扣减。
- 如果请求期望生成多张图片，但只成功产出部分图片，只扣减成功产出的张数。
- 纯文本聊天任务不扣减图片计费单位；后续如需支持文本 token 计费，应作为独立计费单位扩展。

提交前需要做计费预检：

- 如果用户为标准余额制，系统按可用余额判断是否允许提交。
- 如果用户为订阅配额制，系统按剩余配额判断是否允许提交。
- 如果可用余额或剩余配额小于请求可能消耗的最大计费单位，系统应拒绝提交。
- 文生图和图生图的最大可能消耗按请求的 `n` 或内部 `count` 计算。
- `/v1/chat/completions` 图片模式的最大可能消耗按请求的 `n` 计算；未显式指定时按 1 计算。
- 未显式指定数量时按 1 计算。
- 计费预检通过后，实际扣减仍以成功产出数量为准。

标准余额制扣减：

- 提交阶段只做余额预检，不冻结余额。
- 成功阶段按实际产出扣减 `balance`。
- 部分成功时只扣减实际产出点数。
- 失败、取消、超时时不扣减余额。
- 管理员手动增加余额时直接增加 `balance`。
- 管理员手动扣减余额时不得导致 `balance` 小于 0，除非明确允许负余额；本需求默认不允许负余额。

订阅配额制扣减：

- 提交阶段只做剩余配额预检，不冻结配额。
- 交付阶段必须先完成原子扣减，再向用户写入图片结果；扣减失败的图片不得交付。
- 成功阶段按实际交付图片增加 `quota_used`。
- 部分成功时只增加实际产出的 `quota_used`。
- 失败、取消、超时时不增加 `quota_used`。
- 管理员手动增加或扣减当前周期配额时写入 `manual_delta`。
- 订阅周期重置时清零 `quota_used` 和 `manual_delta`，保留 `quota_limit`。
- 未使用完的剩余配额默认不结转到下一周期；如需结转，应作为后续独立需求。

并发场景要求：

- 多个请求同时提交时，提交前预检按当时的可用余额或剩余配额判断。
- 提交前预检只是优化；严格防超卖以交付前原子扣减为准。
- 同一个任务或协议输出的成功扣减必须通过 `charge_key` 幂等，避免重复扣减。

### 6.3 管理端调整

管理员可以对单个用户执行以下通用操作：

- 修改计费类型。
- 设置为无限额度或取消无限额度。
- 可选填写调整原因。
- 查看固定数量的最近调整记录；记录列表使用固定高度滚动区，不能随记录数量无限撑高弹窗。

标准余额制支持：

- 设置当前余额。
- 手动增加余额。
- 手动扣减余额。

订阅配额制支持：

- 设置配额上限。
- 修改订阅周期。
- 重置当前周期配额。
- 清零当前周期已用配额。
- 手动增加当前周期配额。
- 手动扣减当前周期配额。

调整记录需要保留：

- 操作人。
- 目标用户。
- 调整类型。
- 调整时的计费类型。
- 调整前计费状态。
- 调整后计费状态。
- 调整数值。
- 调整原因（可为空）。
- 操作时间。

### 6.4 计费展示

用户管理页需要展示：

- 计费类型：标准余额制或订阅配额制。
- 是否无限额度。
- 今日消耗。
- 近 14 天计费单位消耗曲线。

标准余额制用户还需要展示：

- 当前余额。
- 可用余额。
- 累计消耗。

订阅配额制用户还需要展示：

- 配额上限。
- 已用配额。
- 剩余配额。
- 订阅周期。
- 下次重置时间。

用户详情或编辑弹窗需要支持：

- 修改计费类型。
- 修改当前计费类型对应参数。
- 切换无限额度。
- 手动增减余额或当期配额。
- 查看最近计费调整记录。

普通用户个人资料页需要展示：

- 当前计费类型。
- 标准余额制下的当前余额和可用余额。
- 订阅配额制下的当前周期已用配额、剩余配额、配额上限和下次重置时间。
- 并发额度。
- RPM 限制。

创作页需要展示：

- 当前计费类型。
- 当前可用余额或剩余配额。
- 当前请求预计消耗。
- 余额或配额不足时禁用提交按钮，或在提交后返回明确错误。

### 6.5 API 行为

新增或扩展接口：

- `GET /api/admin/users`：返回每个用户的计费状态字段。
- `GET /api/admin/users/{id}`：返回单个用户计费详情。
- `POST /api/admin/users/{id}`：支持更新计费类型和计费配置字段。
- `POST /api/admin/users/{id}/billing-adjustments`：执行手动增减、清零、重置、类型切换等计费调整。
- `GET /api/admin/users/{id}/billing-adjustments`：查看用户计费调整记录。
- `GET /api/profile` 或现有登录响应：返回当前用户计费状态。

建议返回字段使用统一结构：

```json
{
  "billing": {
    "type": "subscription",
    "unit": "image",
    "unlimited": false,
    "available": 86,
    "standard": null,
    "subscription": {
      "quota_limit": 100,
      "quota_used": 12,
      "manual_delta": 0,
      "quota_period": "monthly",
      "quota_period_started_at": "2026-05-01T00:00:00+08:00",
      "quota_period_ends_at": "2026-06-01T00:00:00+08:00"
    }
  }
}
```

标准余额制余额不足错误：

- HTTP 状态码：`429 Too Many Requests`。
- OpenAI 兼容接口错误类型：`insufficient_quota`。
- 错误码：`user_balance_insufficient`。
- 错误信息：`user balance insufficient`，前端可本地化显示为“用户余额不足”。

示例错误响应：

```json
{
  "error": {
    "message": "user balance insufficient",
    "type": "insufficient_quota",
    "param": null,
    "code": "user_balance_insufficient"
  }
}
```

订阅配额制配额不足错误：

- HTTP 状态码：`429 Too Many Requests`。
- OpenAI 兼容接口错误类型：`insufficient_quota`。
- 错误码：`user_quota_exceeded`。
- 错误信息：`user quota exceeded`，前端可本地化显示为“用户配额不足”。

示例错误响应：

```json
{
  "error": {
    "message": "user quota exceeded",
    "type": "insufficient_quota",
    "param": null,
    "code": "user_quota_exceeded"
  }
}
```

### 6.6 权限

需要新增或复用用户管理权限：

- 查看用户计费状态：拥有“查看用户”权限即可查看。
- 修改用户计费状态：需要“创建/修改用户”权限。
- 查看计费调整记录：拥有“查看用户”权限即可查看。
- 执行计费调整：需要“创建/修改用户”权限。

普通用户只能查看自己的计费状态，不能查看其他用户计费状态，也不能查看调整记录。

### 6.7 日志与审计

计费消费日志需要能回答以下问题：

- 哪个用户消费了计费单位。
- 通过哪个凭据消费。
- 当前使用哪种计费类型。
- 哪个接口或任务消费。
- 哪个模型消费。
- 本次实际扣减了多少计费单位。
- 标准余额制下扣减前后的余额和可用余额。
- 订阅配额制下扣减前后的已用配额和剩余配额。
- 任务 ID 或请求关联 ID。

管理端计费调整需要写入审计日志，并可按用户、计费类型和调整类型筛选。

## 7. 数据需求

建议新增用户计费状态数据，按用户 ID 保存。当前计费类型以 `billing_type` 为准，非当前类型字段只用于审计或回切，不参与扣减。

标准余额制示例：

```json
{
  "user_id": "user_123",
  "billing_type": "standard",
  "unit": "image",
  "unlimited": false,
  "standard": {
    "balance": 100,
    "lifetime_consumed": 12
  },
  "subscription": null,
  "updated_at": "2026-05-08T12:00:00+08:00"
}
```

订阅配额制示例：

```json
{
  "user_id": "user_123",
  "billing_type": "subscription",
  "unit": "image",
  "unlimited": false,
  "standard": null,
  "subscription": {
    "quota_limit": 100,
    "quota_used": 12,
    "manual_delta": 0,
    "quota_period": "monthly",
    "quota_period_started_at": "2026-05-01T00:00:00+08:00",
    "quota_period_ends_at": "2026-06-01T00:00:00+08:00"
  },
  "updated_at": "2026-05-08T12:00:00+08:00"
}
```

建议新增计费调整记录：

```json
{
  "id": "billing_adj_abc",
  "user_id": "user_123",
  "operator_id": "admin",
  "operator_name": "管理员",
  "billing_type": "standard",
  "type": "increase_balance",
  "amount": 10,
  "reason": "测试补偿",
  "before": {
    "billing_type": "standard",
    "balance": 100
  },
  "after": {
    "billing_type": "standard",
    "balance": 110
  },
  "created_at": "2026-05-08T12:00:00+08:00"
}
```

建议新增计费流水或扩展现有日志详情：

```json
{
  "id": "billing_txn_abc",
  "user_id": "user_123",
  "billing_type": "subscription",
  "unit": "image",
  "request_id": "req_abc",
  "task_id": "task_abc",
  "action": "charge",
  "consumed_amount": 3,
  "created_at": "2026-05-08T12:00:00+08:00"
}
```

存储要求：

- SQLite/PostgreSQL/MySQL 后端应通过现有 `json_documents` 或独立表保存用户计费状态。
- JSON 存储后端应保存为独立 JSON 文档。
- 不在 `accounts` 中保存用户余额或订阅配额，避免和上游账号额度混淆。
- 不在日志中作为唯一事实来源计算可用余额或剩余配额，日志只做审计和统计。
- 字段命名需要明确区分余额制和配额制：余额制使用 `balance_*`，订阅配额制使用 `quota_*`。

## 8. 兼容与迁移

- 已存在用户首次读取计费状态时，应创建默认计费状态。
- 已存在用户默认使用全局默认计费类型；未配置时使用标准余额制且余额为 0。
- 部署方如果希望延续“周期额度分配”的使用方式，可以将全局默认计费类型设置为订阅配额制。
- 历史 `quota_used` 统计继续用于报表展示，但不反向扣减新的余额或配额。
- 如果从旧文档或旧实现迁移出 `quota_limit`、`quota_used` 字段，应映射到订阅配额制；旧的预留字段应忽略或清理。
- 管理员、系统任务和账号池刷新任务不受用户计费限制。
- 旧的并发/RPM 配置继续生效，且执行顺序建议为：认证与权限 -> 用户启用状态 -> 计费预检 -> RPM -> 并发等待 -> 上游账号选择。

## 9. 验收标准

### 9.1 管理端

- 管理员可以在用户列表看到计费类型、可用余额/剩余配额、今日消耗和近 14 天消耗。
- 管理员可以给本地用户和 Linuxdo 用户设置独立计费类型。
- 管理员可以将用户设为无限额度。
- 管理员可以对标准余额制用户手动增加或扣减余额，并可选填写原因。
- 管理员可以对订阅配额制用户设置配额上限、重置周期、手动增加或扣减当期配额，并可选填写原因。
- 手动调整选项必须按弹窗中当前选择的计费类型展示：标准余额制只展示余额调整，订阅配额制只展示当期配额调整。
- 如果管理员在弹窗中切换了计费类型但尚未保存计费配置，手动调整区可以跟随新类型展示和选择对应调整项，但执行调整必须禁用并提示先保存计费配置，避免按未落库类型误调余额或当期配额。
- 计费调整后，用户列表和用户详情立即显示最新状态。
- 普通用户无法进入或调用管理员计费管理接口。

### 9.2 用户端

- 普通用户可以看到自己的计费类型。
- 标准余额制用户可以看到当前余额和可用余额。
- 订阅配额制用户可以看到当前周期已用配额、剩余配额和下次重置时间。
- 余额不足时，创作页能明确提示“用户余额不足”。
- 配额不足时，创作页能明确提示“用户配额不足”。
- API 调用方在余额不足时收到 `429` 和 `user_balance_insufficient`。
- API 调用方在配额不足时收到 `429` 和 `user_quota_exceeded`。

### 9.3 后端扣减

- 标准余额制下，成功生成 1 张图片扣减 1 点余额。
- 标准余额制下，成功生成 4 张图片扣减 4 点余额。
- 订阅配额制下，成功生成 1 张图片增加 1 点已用配额。
- 订阅配额制下，成功生成 4 张图片增加 4 点已用配额。
- 失败任务不扣减余额或配额。
- 取消任务不扣减余额或配额。
- 部分成功任务只扣减成功产出数量。
- 同一任务多次完成或取消回调不会重复扣减。
- 高并发成功扣减后余额展示不得为负数。
- 余额或配额不足的请求不会进入上游账号池执行。
- 无限额度用户不触发本地余额不足或配额不足错误。
- 管理员不触发本地用户计费限制。

### 9.4 订阅周期重置

- 每日订阅配额在本地日期 00:00 后首次访问或定时任务执行时重置。
- 每周订阅配额在周一 00:00 后重置。
- 每月订阅配额在每月 1 日 00:00 后重置。
- 重置后 `quota_used` 和 `manual_delta` 清零，`quota_limit` 保持不变。
- 标准余额制不执行周期重置。
- 重置动作写入审计日志或系统日志。

## 11. 风险与约束

- 上游账号额度、本地余额和订阅配额名称容易混淆，界面和接口字段必须明确区分。
- 标准余额制虽然叫“余额”，但当前不包含支付、订单或真实货币结算。
- 严格防超卖依赖本地计费服务在单进程内完成原子扣减；多进程或多实例部署必须使用数据库原子更新和 `charge_key` 唯一约束。
- 服务重启时运行中任务状态可能不完整，未成功产出的任务不扣减。
- 日志统计当前只在成功输出后计算 `quota_used`，不能直接作为扣减事务。
- Linuxdo 用户没有本地密码账号，但仍需要按 `owner_id` 维护计费状态。
- 管理员调整计费状态需要审计，避免无法追溯滥用或误操作。
- 计费类型切换需要留意运行中任务；任务成功时会按结算时用户当前计费类型扣减。

## 12. 建议实施顺序

1. 新增计费服务和存储结构，完成标准余额制与订阅配额制状态初始化、读取、更新、调整记录。
2. 在创作任务提交链路增加计费预检。
3. 在任务成功链路按实际产出扣减，失败和取消不扣减。
4. 扩展 OpenAI 兼容图片接口的计费预检与错误响应。
5. 扩展用户管理接口和个人资料接口返回计费状态。
6. 更新用户管理页、用户详情弹窗和个人资料页。
7. 更新创作页计费状态展示和余额/配额不足提示。
